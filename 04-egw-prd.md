## 产品定位

EGW 是一套面向 **IDC 出站流量管控** 的正向代理网关系统，管理私网到公网的流量出口，实现网络访问控制、出口 IP 管理、流量审计等功能。

**核心愿景：** 让 IDC 出站流量安全、可控、可观测。

**与 AGW 的关系：**
```
互联网 ←→ AGW（反向代理）←→ 私网 ←→ EGW（正向代理）←→ 互联网
           入站 (Ingress)                  出站 (Egress)
```

## 用户场景

### 场景 1：IDC 爬虫出口管控
- 多个爬虫业务需要访问互联网
- 通过 EGW 统一管理出口 IP
- 不同业务分配不同出口 IP 池
- 防止单一 IP 被封

### 场景 2：企业安全出站
- 员工访问互联网需经过 EGW
- 按部门设置不同的访问策略
- 记录所有出站访问日志
- 敏感域名拦截

### 场景 3：AI Agent 外部调用
- AI Agent 调用外部 API（LLM、搜索等）
- 通过 EGW 统一管控出口流量
- 监控 API 调用成本和频率
- 按业务设置不同的限流策略

### 场景 4：多区域出口调度
- 在不同国家部署出口节点
- 按用户/业务自动选择最优出口
- 故障自动切换到备用出口

### 场景 5：随机出口
- 业务对出口 IP 无要求
- EGW 自动从可用 IP 池随机选择
- 负载均衡到各出口节点

### 场景 6：固定 IP/段出口
- 对方需要知道我们的来源 IP 或 IP 段
- 绑定固定 IP 出口，保证一致性
- 支持 IP 段（A 段 / B 段 / C 段）级别分配
- 适用于：API 对接、白名单合作、合作伙伴

### 场景 7：爬虫多 IP 出口
- 单台机器绑定大量 IP（数百个）
- 按策略轮换 IP，防止单 IP 被封
- IP 使用频率控制（每个 IP 间隔多久再用）
- IP 生命管理（被封自动剔除，恢复自动加回）
- 按域名/IP 段绑定不同出口策略

### 场景 8：Docker 集群部署
- EGW 实例运行在 Docker 集群上
- 不同集群节点属性有差异：
  - 所在机房/区域
  - 绑定的 IP 段不同
  - 带宽/性能不同
  - 用途不同（爬虫/API/AI 等）
- EGW 需要感知集群拓扑
- 按节点属性智能调度

### 场景 9：边缘节点数据回拉
- DC（数据中心）需要从边缘节点拉取数据
- 流量路径：DC → EGW → 边缘节点 → 拉取数据 → DC
- 应用场景：
  - CDN 回源拉取
  - 边缘计算结果回传
  - 远程备份拉取
  - 分布式存储数据同步
- 需要管控：
  - 回拉带宽控制（避免占满出口带宽）
  - 回拉频率限制
  - 回拉数据量统计
  - 源站认证（防止未授权回拉）
  - 大文件传输优化（断点续传、分片）

## 技术方案

### 架构设计

**数据面（Data Plane）**
- 语言：Rust（参考 G3）
- 部署：Kubernetes（运行在 K8s 集群上）
- 职责：高性能代理、协议解析、流量转发、出口调度
- 目标：极致性能和低延迟

**控制面（Control Plane）**
- 语言：Go（与 AGW 统一技术栈）
- 数据库：PostgreSQL + Redis
- 职责：配置管理、出口 IP 池管理、用户管理、监控聚合、审计日志
- 部署：Kubernetes

**与 AGW 共享：**
- 统一控制面技术栈（Go + PG + Redis）
- 统一认证体系
- 统一监控面板
- 统一配置管理

### 技术选型要点
- 协议：HTTP/1.1、HTTP/2、SOCKS5、TLS
- 代理模式：正向代理、透明代理、代理链
- 监控：Prometheus + Grafana + StatsD
- 配置：YAML + 动态热加载
- 部署：Docker / Kubernetes

### 候选方案

| 方案 | 说明 | 优缺点 |
|------|------|--------|
| **A. 直接用 G3** | 部署 G3，通过配置实现需求 | 快速，但缺少 AI/IoT 能力 |
| **B. 基于 G3 二开** | Fork G3，添加自定义功能 | 有基础，但 Rust 二次开发成本高 |
| **C. 自研 + 参考 G3** | 自研控制面，参考 G3 数据面设计 | 灵活，但开发周期长 |
| **D. Go 自研** | 全部用 Go 自研 | 与 AGW 技术栈统一，但性能不如 Rust |
